# UI / UX

本製品に画面（GUI）はない。ここでの「UI」は、**非エンジニアが対話とファイルを通じて受け取る体験**を指す。
体験の良し悪しがそのまま製品価値になるため、以下を全スプリント共通の非機能要件とする。

## 対象読者の再定義（2026-07-08 ユーザー指示）

対象は「**技術に多少関心のある非エンジニア**」。Claude Code をわざわざ使う層であり、一般的な技術用語はむしろ知っておきたいと考えている。したがって**過度な平易化はしない**。一般に通じる技術用語は積極的に使い、無理に言い換えたり幼稚なメタファーに置き換えたりしない（かえって分かりにくくなるため）。

## 体験の原則

1. **わかりやすい標準語彙**: ユーザーに見せる文言は、日常語をベースにしつつ**一般に通じる技術用語はそのまま使う**。無理な言い換え・幼稚化をしない。
   - **そのまま使う語（言い換え不要）**: ディレクトリ、フォルダ、ファイル、インストール、コミット、プラグイン、コネクタ、TODO、カレンダー、メール、リンク、設定 など。
   - **簡潔な補足を1回だけ添える語（初出のみ、本当に馴染みの薄い語に限定）**: OAuth、リポジトリ、frontmatter、MCP、トークン、環境変数 など。例: 「OAuth（アプリ同士を安全につなぐ仕組み）」。毎回は繰り返さない。
   - **幼稚なメタファーの禁止**: 「秘書の家」等の比喩で技術概念を置き換えない。`secretary/` は「**秘書ディレクトリ**」または「**秘書フォルダ**」と呼ぶ。
2. **3行報告**: 秘書の報告は原則3行以内。「やったこと」「結果」「次に何が起きるか」を一言ずつ。専門用語の扱いは原則1に従う（一般語はそのまま、馴染みの薄い語だけ初出補足）。長い詳細は聞かれたら出す。
3. **進行が見える**: 複数手順の作業では「いま計画→道具→確認→結果のどこにいるか」を毎回宣言する（第1回座学の実況語彙。これは**段階の呼び名**であり、技術用語を避ける意味ではない）。
4. **不安を先回り**: 時間がかかる・確認が要る・失敗しうる箇所を、始める前に一言で予告する。
5. **選択はわかりやすく**: 質問は2〜3択、それぞれに具体例。抽象的な専門語だけの問い（「認証方式は？」）は避け、選択肢を具体的にする（「見る人を制限しますか？ 誰でも見られる／招待した人だけ」）。ただし通じる技術用語は使ってよい。
6. **英語のエラーを翻訳**: コネクタや git の英語エラーは、そのまま出さず「何が起きて・どうすれば直るか」に言い換えてから示す（原文を併記するのは可）。

## インストール〜初回体験

- インストールは cc-company と同じ**3コマンド**。各コマンドの前に「今から何をするか」を1行で説明する。
- 初回 `/secretary` は、いきなり機能一覧を出さず、**やさしい数問**（呼び方・主に使うサービス・秘書に任せたいこと 程度）から始める。
- オンボーディング完了時に「秘書ディレクトリ（`secretary/`）を作成しました」と、生成された場所を具体的に示し、最初のコミットが済んだことを伝える（「秘書の家」等の比喩は使わない）。

## 秘書の口調

- 敬語ベースだが硬すぎない。オーナーの `preferences.md` に呼び方・口調の希望があればそれに従う。
- 断定できないことは断定しない。根拠がないことは「確認します」と述べる。

## SKILL のロード体験（内部だが体験に影響）

- ルーター SKILL.md は薄く保ち、必要になった機能だけを段階的に読み込む。起動時に全機能を読み込んで重くしない。
- モード判定（今日やること／記憶ケア／接続設定／開発）を、ユーザーの自然な言い回しから拾えるようにする。

## 非機能要件

- **可搬性**: macOS / Windows いずれの受講者環境でも、生成されるワークスペースとコマンドが破綻しない。
- **決定性**: オンボーディングが生成するファイル構造は、同じ回答なら毎回同じになる（検証可能性のため）。
- **回復性**: 途中で中断しても、再起動しおり（`_resume.md`）で文脈を復元して続けられる。
- **安全性**: 破壊的操作の前に必ず確認。資格情報を書き込まない・コミットしない。

## アクセシビリティ（言語）

- すべてのユーザー向け出力は日本語。技術識別子（コマンド名・ファイル名）は原形のまま示す。一般に通じる語はそのまま、本当に馴染みの薄い語だけ初出で簡潔に補足する（原則1）。
