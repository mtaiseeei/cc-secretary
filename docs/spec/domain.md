# Domain

秘書プラグインが扱う概念・データ・業務ルール。「実行アプリのDB」ではなく、
**生成されるワークスペースのファイル構造**と**秘書の振る舞いを決めるルール**がドメインの中心。

## 主要概念

| 概念 | 意味 | 置き場 |
|---|---|---|
| 秘書（Secretary） | ユーザーの代わりに記憶・整理・下調べ・成果物作成・開発を行う AI 役割 | プラグインの skills 群 |
| 秘書の家（Workspace） | オンボーディングで生成する `secretary/` ディレクトリ。git 管理される | ユーザーのローカル |
| 記憶（Memory） | 決定・好み・進行中案件。ローカルに蓄積し保護する | `secretary/memory/` |
| 成果物（Deliverable） | 企画書・調査まとめ等の文書。**正本はローカル** | `secretary/docs/` |
| 外部データ（External） | メール・予定・ファイル・タスク。SaaS に置いたまま参照 | 各コネクタ（同期しない） |
| 道具箱（Toolbox） | 秘書が使える機能スキル群（daily / memory-care / build 等） | プラグイン skills |

## 生成されるワークスペース構造（正本: DESIGN.md）

```
secretary/                  ← 秘書の家（初回セットアップで git init）
├── AGENTS.md               ← 指示の正本（スコープ・根拠・出力規約・記憶保護・自動コミット・報告の型）
├── CLAUDE.md               ← AGENTS.md へのポインタのみ
├── inbox/                  ← 走り書き・下書き・クイックキャプチャ
├── docs/                   ← 成果物の正本（YYYY/MM/YYYY-MM-DD_<title>.md）
├── projects/               ← 進行中案件ごとのフォルダ（軽量。案件指示は AGENTS.md）
└── memory/
    ├── MEMORY.md           ← 記憶の目次（1行索引。cc-company から継承）
    ├── decisions/          ← 決定ログ（YYYY-MM-DD-decisions.md）
    └── preferences.md      ← オーナーの好み・口調・環境情報
```

- `10_sources/` に相当する層は**存在しない**。
- 外部データの根拠は「**サービス名＋URL/ID＋日付**」で引用する。本文はローカルに保存しない。

## 記憶（Memory）のルール

- **MEMORY.md は索引**: 1行1記憶（`- [タイトル](ファイル) — ひとこと`）。記憶ファイルを増減したら索引を必ず更新する。
- **決定ログ**: `memory/decisions/YYYY-MM-DD-decisions.md`。確定した意思決定を日付ごとに追記。
- **好み**: `memory/preferences.md`。口調・呼び方・作業環境・避けたいこと。
- **保護規則**（不変条件）:
  - 空内容・実質空で既存記憶を上書きしない。
  - 削除の前に「何を消すか」を非エンジニアにわかる言葉で警告し、確認を取る。
  - 相対日付は絶対日付に変換して記録する。
- **オンデマンド振り返り**: 「振り返りして」で、進行中案件・決定を棚卸しし、記憶を整理する（自動の `case-NNN` 生成はしない）。

## 出力規約（成果物）

- ファイル名: `YYYY-MM-DD_<title>.md`。保存先は `secretary/docs/YYYY/MM/`。
- frontmatter 必須: `createdAt`（`YYYY-MM-DD HH:mm`）、`tags`。
- 1ファイル1トピック。見出しに固有名詞を入れる（後から検索できるように）。

## 自動コミットのルール

- 作業の節目（オンボーディング完了・成果物作成・記憶更新・案件の区切り）ごとに、`secretary/` 内でローカル commit。
- コミットメッセージは**日本語**、何をしたかが一目でわかる粒度。
- **push はしない**（明示指示時のみ）。

## コネクタ（外部データ参照）

| 区分 | 対象 | 接続方法 |
|---|---|---|
| 第一級 | Gmail / Google Calendar / Google Drive / Microsoft 365 | claude.ai 公式コネクタ（設定画面から OAuth 自動） |
| 任意 | Notion | `mcp.notion.com`（OAuth 自動） |
| 初期見送り | Chatwork / LINE 等の国内チャット | 公式リモート MCP がなく自作ラッパーが必要 |

- cc-company の Google Cloud Console 手作業は**ほぼ消える**（公式コネクタで代替）。
- 接続診断は「**実エラーで原因を確定してから案内**」する（cc-company の診断手順を継承）。

## 秘書の主要ふるまい（機能ドメイン）

- **オンボーディング**: 初回3問程度 → ワークスペース生成 → `git init` → 最初のコミット。
- **今日やること（daily）**: コネクタから予定・タスクを参照し、TODO と突き合わせて、根拠つき3行で提示。
- **記憶ケア（memory-care）**: 記憶の保護・復元・オンデマンド振り返り・再起動しおり（`_resume.md`）。
- **コネクタ接続ガイド（setup-google / setup-microsoft）**: 接続手順の案内＋接続確認テスト。
- **開発（build）**: 開発依頼を「やさしいハーネス」に接続。

## テンプレートの配置とパス解決（確定）

オンボーディングが実体化する雛形（`AGENTS.md` / `CLAUDE.md` / `memory/MEMORY.md` / `memory/preferences.md` の初期形）は、**配布プラグインの実体に必ず同梱される場所**に置く。

- **配置**: `plugins/cc-secretary/templates/`（リポジトリ直下 `templates/` ではない）。
  - 理由: marketplace の source はプラグインディレクトリ `plugins/cc-secretary/`。リポジトリ直下に置くと、実インストール時に雛形がプラグイン一式に含まれず、onboarding が参照できない恐れがある。
- **参照方法**: SKILL からの雛形参照は **`${CLAUDE_PLUGIN_ROOT}` 相対**に統一する（`${CLAUDE_PLUGIN_ROOT}/templates/...`）。`CLAUDE_PLUGIN_ROOT` はプラグインのインストール先を指す環境変数で、開発時のリポジトリ配置と実インストール先の両方で正しく解決される。
- **生成先との区別**: これは「プラグイン側の雛形（読み取り元）」の話。実体化先は常にユーザーの `secretary/`（上記ワークスペース構造）。
- 回帰スクリプトも、この配置・参照パスに追従して検証する。

DESIGN.md の構成図は方針であり、ディレクトリ配置の厳密な正本はこの spec 側で確定する。
