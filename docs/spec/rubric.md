# Evaluation Rubric

## プロジェクト種別

**Claude Code プラグイン（マークダウンのスキル群＋テンプレート＋ルールファイル）**。
Web アプリ・サーバー・GUI ではない。したがって Evaluator は「ブラウザで動かして採点」する Web アプリ用の既定基準を**そのまま使わない**。
評価は **(1) 静的検証（マニフェスト・スキル構文・参照整合）**、**(2) スクリプト化したドライラン（生成物・保護規則の実証）**、**(3) 内容ルール検査（非エンジニア文言・不変条件）** を軸にする。

## 検証方法（証跡の取り方）

Evaluator は以下を実施し、実行コマンドと結果を `docs/feedback/sprint-*.md` に必ず残す。証跡のない合格は無効。

1. **マニフェスト検証**: `.claude-plugin/marketplace.json` と `plugin.json` が有効な JSON で、必須フィールドが揃い、参照する skills/agents/rules のパスが実在する（`jq . <file>` や `python3 -m json.tool`、`test -e` で確認）。
2. **スキル構文検証**: 各 `SKILL.md` の frontmatter（`name` / `description` 等）が有効で、`name` が一意。段階ロードの参照先ファイルが実在する。
3. **ドライラン（生成物の実証）**: Generator が用意した回帰スクリプト（シェル/Python 等、対話 Claude に依存しない）を実行し、
   - オンボーディングのテンプレート実体化が `docs/spec/domain.md` のワークスペース構造どおりの木を生成する、
   - 記憶保護（空上書き拒否・削除前警告・MEMORY.md 索引更新）が期待どおり振る舞う、
   等を assert する。文字列の完全一致ではなく、**構造・振る舞い**を検証する。
4. **内容ルール検査**: ユーザー向け文言に生の英語エラーが露出していないか、馴染みの薄い語（OAuth 等）が初出補足なしで使われていないか、幼稚なメタファー（「秘書の家」等）が残っていないか、報告3行型ルールと `rules/plain-language.md` が各エージェント/スキルから参照されているかを grep ベースで確認する（語彙方針は改訂 `docs/spec/ui.md`）。
5. **不変条件検査**: `docs/spec/constraints.md` の絶対ルールに反する記述・実装（`agentic-harness` への書き込み、同期層の追加、push 自動化、資格情報の保存/コミット）が**ないこと**を確認する。
6. **手動ライブ確認（可能なとき）**: サインイン済み Claude Code セッションが使える場合のみ、実際にプラグインを入れて `/secretary` を走らせるチェックリストを実施し記録する。使えない場合はスクリプト検証をゲートとし、手動確認は「未実施の手動確認」として明記する（Web アプリのスクショ必須要件は本種別には適用しない）。

## 採点基準と閾値

各基準を 0〜5 で採点する。**1つでも閾値を下回ればそのスプリントは不合格**。

| # | 基準 | 見るもの | 合格閾値 |
|---|---|---|---|
| C1 | 完成度 | 対象スプリント契約の受入基準をすべて満たすか | ≥ 4 |
| C2 | 構文・整合 | マニフェスト/スキルが有効・参照パスが実在・`name` 一意 | 5（欠陥ゼロ） |
| C3 | 機能の実証 | ドライランで生成物・保護規則が期待どおり振る舞う | ≥ 4 |
| C4 | 非エンジニア体験 | 日常語＋具体例・3行報告・進行の可視化・生英語エラー非露出 | ≥ 4 |
| C5 | 安全・規律 | constraints.md の絶対ルール・不変条件に違反しない | 5（違反ゼロ） |
| C6 | 無回帰 | 既存スプリントの受入を保護する回帰チェックがパス | 5（新規失敗ゼロ） |

- C2・C5・C6 は**ゼロ許容**（1件でも欠陥/違反/新規失敗があれば不合格）。
- C1・C3・C4 は 4 以上で合格。3 以下は不合格とし、具体的な再現手順とともに Generator へ差し戻す。

## スコアのアンカー例

**C4（非エンジニア体験）**（語彙方針は改訂 `docs/spec/ui.md`）
- 5: ユーザー向け文言が「わかりやすい標準語彙」を徹底。一般技術用語はそのまま使い、馴染みの薄い語のみ初出補足。幼稚なメタファー（「秘書の家」等）ゼロ。報告は3行型で「次に何が起きるか」が必ずある。英語エラーは翻訳済み。進行が「計画→道具→確認→結果」で宣言される。
- 4: 概ね守られるが、馴染みの薄い語の初出補足漏れ等が1〜2箇所。体験は成立している。
- 3: 報告が長い／過度な平易化や幼稚なメタファーが複数／馴染みの薄い語が無補足で頻出／進行が見えない。→ 不合格。
- 1: 英語のまま・技術者向けの説明が中心、または幼稚な言い換えで概念が伝わらない。対象ユーザーが使えない。

**C3（機能の実証）**
- 5: ドライランが生成物構造・記憶保護・出力規約をすべて assert しパス。証跡（コマンドと出力）が揃う。
- 4: 主要な振る舞いは assert・パス。周辺の一部が手動確認どまり（理由を明記）。
- 3: スクリプトが未整備で目視のみ、または一部の assert が落ちる。→ 不合格。

**C2（構文・整合）**
- 5: 全マニフェスト/スキルが有効、参照パス全実在、`name` 一意。
- 4以下: JSON 破損・欠落フィールド・デッドリンク・`name` 重複のいずれか1つ以上。→ 不合格（ゼロ許容）。

## 差し戻しの分類

- **implementation-issue**: 実装の欠陥（Generator へ差し戻し）。
- **spec-issue**: 受入基準・仕様の矛盾/欠落（オーケストレーター経由で Planner へ）。
- Evaluator は rubric の変更を feedback で**提案**できるが、適用するのは Planner だけ。

## 適用外にする Web アプリ既定基準

- スクリーンショット必須・レスポンシブ検証・ビジュアルデザイン採点・エンドポイント/DB 状態確認は、本製品には**適用しない**（該当する実行 UI が存在しないため）。GUI を伴う手動ライブ確認が可能な場合のみ、補助証跡として記録する。
